핵심 요약
  • OAuth는 비밀번호를 공유하지 않고도 제3자 앱이 제한된 권한으로 API에 접근하게 하는 권한 위임(authorization) 프레임워크다.
  • 권한은 토큰으로 표현되며, 토큰은 스코프(scope)로 범위를 제한하고 만료/회수가 가능하다.
  • GitHub Pages 배포는 보통 “Pages에 직접 publish”라기보다, 리포지토리 푸시/워크플로 트리거를 안전하게 자동화하는 문제로 귀결된다.

1) 배경과 문제의식

“OAuth로 GitHub.io에 publish” 같은 요구는 대개 인증(Authentication)과 권한 부여(Authorization)가 섞여 나온 표현이다. 자동 배포(CI/CD)나 외부 서비스 연동에서는 결국 GitHub API 또는 리포지토리 접근이 필요한데, 그 접근을 비밀번호 공유 없이, 필요 최소 권한으로, 만료·회수 가능하게 다루는 게 핵심 과제다.

OAuth는 이 과제를 해결하기 위해 “권한을 토큰으로 쪼개서 전달”하는 방식으로 표준화되었다.

2) 제1원리 (First Principles)

정의(Definition)

OAuth 2.0은 Resource Owner(사용자)가 Client(앱)에게 Authorization Server를 통해 Resource Server(API) 접근 권한을 토큰으로 위임하는 프레임워크다.

직관(Intuition)

  • 비밀번호는 “만능키”라서 공유하는 순간 위험이 폭발한다.
  • 대신 “문 하나만 열 수 있는, 시간 제한이 있는 열쇠(Access Token)”를 주면 피해 범위를 통제할 수 있다.

결과(Implication)

  • 토큰은 scope로 권한 범위를 제한할 수 있다.
  • 토큰은 expiration으로 자동 만료될 수 있다.
  • 토큰은 revocation으로 회수 가능하다.

3) 대조와 비교

실무에서 OAuth와 함께 자주 비교되는 접근 방식들은 다음과 같다.

  • PAT(Personal Access Token): 설정은 쉽지만 “내 계정 장기 키”에 가까워 유출 시 파급이 크다.
  • GitHub App: 설치(installation) 기반 최소권한·감사에 유리하지만 초기 설계가 더 필요하다.
  • GitHub Actions의 GITHUB_TOKEN: 워크플로 실행 중에만 유효한 단기 토큰. Pages 배포에 표준적으로 많이 쓰인다.
  • Deploy key(SSH): 리포 단위로 격리 가능하지만, API 스코프 모델과는 결이 다르다.

4) 맥락적 연결 (GitHub Pages/배포 자동화 관점)

상황 A: GitHub Actions로 GitHub Pages 배포

이 경우 OAuth를 직접 구현하기보다, Actions에서 제공하는 GITHUB_TOKEN을 활용하는 게 일반적으로 더 안전하고 단순하다. 핵심은 워크플로가 Pages에 배포할 수 있도록 권한을 명시하는 것이다.

Example (concept): GitHub Actions permissions for Pages
permissions:
  contents: read
  pages: write
  id-token: write

상황 B: 외부 서비스/로컬 툴이 GitHub에 접근해 배포를 유도

이때 “OAuth publish”라는 표현이 흔하지만, 실체는 보통 아래 중 하나로 귀결된다.

  • 리포지토리에 커밋/푸시(배포 트리거)
  • GitHub API로 워크플로(예: pages deploy)를 트리거

즉, Pages 자체에 직접 ‘업로드’한다기보다, GitHub가 신뢰하는 경로(리포/워크플로)를 통해 배포가 진행된다.

5) 용어 사전

  • OAuth 2.0: 비밀번호 공유 없이 토큰으로 권한을 위임하는 표준 프레임워크.
  • Authentication(인증): “너 누구냐?”를 확인하는 것(로그인).
  • Authorization(권한 부여): “이 작업을 해도 되냐?”를 결정하는 것.
  • Access Token: API 호출에 쓰는 제한 권한·단기 열쇠.
  • Scope: 토큰이 허용하는 행위의 범위(예: repo 읽기/쓰기).
  • Expiration: 토큰이 자동으로 만료되는 속성.
  • Revocation: 발급된 토큰을 강제로 회수하는 것.
  • OAuth App: 사용자 동의로 토큰을 받아 GitHub API에 접근하는 앱 형태.
  • GitHub App: 설치 기반 최소권한 모델(리포/조직 단위 제어가 강점).
  • GITHUB_TOKEN: GitHub Actions 실행 중 자동 발급되는 임시 토큰.
  • GitHub Pages: 정적 사이트 호스팅 기능. 보통 빌드 산출물을 배포해 서비스한다.
  • CI/CD: 빌드·테스트·배포를 자동화하는 파이프라인.
  • PAT: 개인 계정에 귀속되는 액세스 토큰(운영/보안 주의 필요).
  • Deploy key: 특정 리포에 연결된 SSH 키(배포·푸시 등에 사용).
